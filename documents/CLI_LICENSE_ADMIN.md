# FreeRangeNotify Licensing & Admin CLI — Design Document

## The Two Business Scenarios

### Scenario A: We host it (hosted / SaaS)

We run FreeRangeNotify on our own infrastructure at freerangenotify.com.

- User signs up → gets 1-month free trial (already works today).
- Trial ends → user must buy a subscription (payment gateway — future).
- **Admin need:** We sometimes want to exempt a user from paying and just grant them a subscription from our backend. This is the `frn renew-license` CLI use case.
- **Admin need:** We sometimes want to delete a user's account from the backend. This is the `frn delete-account` CLI use case.
- License verification happens **locally** — the server checks its own Elasticsearch `subscriptions` index.

### Scenario B: Customer hosts it (self-hosted / on-prem)

A customer installs FreeRangeNotify on their own servers.

- During installation, they buy a license key from us.
- They attach that license key using `frn license attach --license-key <key>`.
- License verification does **NOT happen locally**. The customer's server calls **our central endpoint** (e.g. `https://api.freerangenotify.com/v1/license/verify`) to confirm the key is valid and not expired.
- When the license expires, they buy a new key from us.
- **The customer cannot grant themselves free subscriptions or bypass licensing.** The subscription management APIs must not exist in self-hosted mode.

## What Gets Shipped to Self-Hosted Customers

**We do NOT give self-hosted customers Docker images, docker-compose files, or any env-based config.**

If we gave them Docker/compose, they could set `FREERANGE_LICENSING_ENABLED=false` and use the product for free forever. That's the whole reason for this design.

### Self-hosted distribution flow

```
Customer's machine
──────────────────

Step 1: Install the frn CLI
  $ curl -fsSL https://get.freerangenotify.com | sh
  (or: download frn binary from releases page)

Step 2: Run frn install
  $ frn install

  The installer:
  ├── Prompts for Elasticsearch URL, username, password
  ├── Prompts for Redis host, port, password
  ├── Prompts for license key (purchased from us)
  ├── Validates ES connectivity
  ├── Validates Redis connectivity
  ├── Validates license key against our server (https://api.freerangenotify.com/v1/license/verify)
  ├── If all valid:
  │   ├── Downloads pre-built `server` and `worker` binaries from our CDN
  │   ├── Writes config.yaml with ES/Redis creds + license key (NOT .env, not compose)
  │   ├── Creates systemd service files (freerange-server.service, freerange-worker.service)
  │   ├── Starts the services
  │   └── Runs health check
  └── If invalid: stops and shows error
```

### What the customer gets

| Item | Delivered? | Notes |
|---|---|---|
| `server` binary | Yes | Downloaded by `frn install` |
| `worker` binary | Yes | Downloaded by `frn install` |
| `frn` CLI | Yes | Installed first, used for setup + license management |
| `config.yaml` | Yes | Generated by installer, contains ES/Redis creds + license key |
| systemd service files | Yes | Generated by installer |
| Docker image | **No** | Not shipped for self-hosted |
| docker-compose.yml | **No** | Not shipped for self-hosted |
| .env file | **No** | Not used for self-hosted |
| Source code | Available (open source) | Legal terms prohibit removing license checks |

### Why this is secure

1. **Licensing config is compiled into the binary** — The self-hosted build uses Go build tags (`//go:build selfhosted`) to hardcode `licensing.enabled=true` and `deployment_mode=self_hosted` at compile time. The customer cannot override these values via config.yaml, env vars, or any external file. They are constants in the binary.
2. **Two separate builds** — We produce two binaries from the same codebase:
   - `go build -tags selfhosted` → self-hosted binary (licensing always on, mode locked to `self_hosted`)
   - `go build` (no tag) → hosted binary (licensing config from env/config as usual)
3. **License validated before install** — `frn install` calls our server to verify the license key before downloading anything. No valid license = no binaries.
4. **Binaries are pre-built** — Customer doesn't compile from source (though they could since it's open source — that's a legal concern, not a technical one).
5. **Config.yaml only contains infra creds** — ES, Redis, SMTP, license key. The licensing on/off switch is NOT in config.yaml — it's baked into the binary.

### How build tags work

```go
// internal/config/licensing_default.go
// Used for hosted builds (default, no build tag)

//go:build !selfhosted

package config

// LicensingOverrides returns nil — all licensing config comes from config.yaml / env vars.
func LicensingOverrides() *LicensingConfig {
    return nil
}
```

```go
// internal/config/licensing_selfhosted.go
// Used for self-hosted builds only

//go:build selfhosted

package config

// LicensingOverrides returns hardcoded licensing config that cannot be changed.
// These values override anything in config.yaml or environment variables.
func LicensingOverrides() *LicensingConfig {
    return &LicensingConfig{
        Enabled:        true,
        DeploymentMode: "self_hosted",
        FailMode:       "fail_closed",
    }
}
```

```go
// In config.go Load(), after unmarshalling:
if overrides := LicensingOverrides(); overrides != nil {
    config.Licensing.Enabled = overrides.Enabled
    config.Licensing.DeploymentMode = overrides.DeploymentMode
    config.Licensing.FailMode = overrides.FailMode
}
```

**What the customer CAN change in config.yaml:**
- Elasticsearch URL, username, password
- Redis host, port, password
- SMTP settings
- License key (to renew/replace their key)
- Feature flags (workflows, digests, etc.)

**What the customer CANNOT change (baked into binary):**
- `licensing.enabled` — always `true`
- `licensing.deployment_mode` — always `self_hosted`
- `licensing.fail_mode` — always `fail_closed`
- Ops routes — not registered (route table is build-tag gated too)

### What about Docker for self-hosted? (why not)

| Docker approach | Problem |
|---|---|
| Ship Docker image + compose | Customer sets `FREERANGE_LICENSING_ENABLED=false` in env → free forever |
| Ship Docker image, no compose | Customer writes their own compose with env override → same problem |
| Ship Docker image with licensing baked in | Customer extracts binary, writes own config.yaml with `licensing.enabled: false` → free forever |

**With build tags, even the extracted binary is safe** — `licensing.enabled=true` is a compiled constant, not a config value. But Docker still makes it too easy to manipulate the runtime environment, so we ship raw binaries + systemd instead.

## Security Audit: Existing Loopholes

### Loophole 1: Subscription CRUD is unprotected (CRITICAL)

Today in `routes.go:322-329`, these endpoints are always registered and only need a regular user JWT:

```
POST   /v1/admin/licensing/subscriptions      ← any logged-in user can create a subscription
PUT    /v1/admin/licensing/subscriptions/:id   ← any logged-in user can extend one
GET    /v1/admin/licensing/subscriptions       ← any logged-in user can list all
```

**Impact (hosted):** Any registered user calls `POST /v1/admin/licensing/subscriptions` with their own `tenant_id`, sets `plan: "enterprise"` and `current_period_end: "2099-12-31"`. They never pay.

**Impact (self-hosted):** Customer creates a fake subscription in ES. If they configure `deployment_mode: hosted` (just an env var change), the `HostedChecker` would validate against it → free forever.

**Fix:** These subscription CRUD endpoints must be moved behind ops auth + hosted-only gating. Regular users should only have read access via `/v1/billing/subscription`.

### Loophole 2: Customer can disable licensing entirely (CLOSED by build tags)

Previously: customer sets `FREERANGE_LICENSING_ENABLED=false` in `.env` → NoopChecker → free forever.

**Now closed at compile time:** Self-hosted binary is built with `go build -tags selfhosted`. The `LicensingOverrides()` function returns hardcoded `Enabled=true`, `DeploymentMode=self_hosted`, `FailMode=fail_closed`. These are Go constants compiled into the binary — no config file, env var, or runtime flag can change them.

**Additionally:**
- No Docker/compose/.env shipped — systemd + raw binaries only
- `frn install` validates license before downloading binaries
- Binary phones home periodically (heartbeat)

**Remaining risk:** Customer rebuilds from source code (it's open source) with build tags removed.
**Mitigation:** Legal terms in license agreement + heartbeat detects missing instances.

### Loophole 3: Customer can switch deployment_mode (MEDIUM)

A self-hosted customer could change `FREERANGE_LICENSING_DEPLOYMENT_MODE=hosted` in their env, then create fake subscriptions in their local ES. The `HostedChecker` would accept them.

**Fix:** In self-hosted builds, `deployment_mode` should be hardcoded or validated against the license key claims. The license JWT should contain a `mode` claim. If the license says `self_hosted`, the server refuses to start in `hosted` mode.

### Summary of loopholes

| # | Loophole | Severity | Status | Fix |
|---|---|---|---|---|
| 1 | Subscription CRUD open to any JWT user | CRITICAL | Open | Move to ops-auth, hosted-only |
| 2 | Customer sets `licensing.enabled=false` | MEDIUM | **Closed** | Build tags: `LicensingOverrides()` compiled into binary. No config/env override possible. |
| 3 | Customer switches to `hosted` mode | MEDIUM | Open | Validate mode against license key claims |

**Loopholes 1 and 3 must be addressed before shipping ops endpoints.**

## Proposed Design

### Principle 1: Separate the two modes completely

| | Hosted (Scenario A) | Self-Hosted (Scenario B) |
|---|---|---|
| **Who grants access?** | Us (the platform operators) | Us (via signed license key) |
| **Where is the check?** | Local Elasticsearch | Remote call to our licensing server |
| **Can user grant themselves access?** | No | No |
| **Subscription CRUD APIs available?** | Yes, but ops-only | **No, completely disabled** |

### Principle 2: Ops-only authentication for dangerous endpoints

Instead of regular JWT auth, dangerous admin endpoints get a separate **ops token** system:

- A static secret stored in server config: `security.ops_secret` (env: `FREERANGE_OPS_SECRET`).
- The CLI sends this secret as `Authorization: Bearer ops:<secret>` header.
- A new middleware (`OpsAuth`) validates this header. It does NOT use JWT or user sessions.
- This secret is known only to the infrastructure operator (us in hosted, nobody in self-hosted since the routes are disabled).

**Why not JWT?**
- JWT tokens belong to regular users. Any user can get one.
- Ops actions must only be available to the person who controls the server config, not to any registered user.

### Principle 3: Deployment-mode gating at route registration

Routes are registered or skipped based on `licensing.deployment_mode`:

```
setupAdminRoutes():
  if deployment_mode == "hosted":
    register ops routes (license renew, user delete)
  else:
    skip ops routes entirely — they don't exist in the binary's route table
```

A self-hosted customer cannot call these endpoints even if they know the URL — the route handler is never registered. This is the strongest possible protection.

### Principle 4: Self-hosted license verification calls our server

Today, `SelfHostedChecker` only validates the JWT signature locally. A customer could create their own key pair and sign their own license.

**Fix:** Add periodic remote verification.

- `SelfHostedChecker` sends the license key to `https://api.freerangenotify.com/v1/license/verify` every N minutes (configurable via `licensing.self_hosted.verify_interval_seconds`, default 300).
- Our server validates the key against our private records (active customer, not revoked, correct plan).
- If verification fails or is unreachable, use `fail_mode` setting (grace window or block).
- The public key PEM is fetched from our server on first startup, not bundled in the customer's config. This prevents the customer from using their own key pair.

### Principle 5: Lock existing subscription CRUD behind ops auth

The current endpoints at `/v1/admin/licensing/subscriptions` (POST, PUT) must be moved behind ops auth. They are the biggest existing loophole.

- `POST /v1/admin/licensing/subscriptions` → move to `/v1/ops/subscriptions` (ops auth only, hosted only)
- `PUT /v1/admin/licensing/subscriptions/:id` → move to `/v1/ops/subscriptions/:id` (ops auth only, hosted only)
- `GET /v1/admin/licensing/subscriptions` → keep for read-only audit (JWT auth ok)
- `GET /v1/admin/licensing/subscriptions/:id` → keep for read-only audit (JWT auth ok)

### Principle 6: Validate deployment mode against license claims

The self-hosted license JWT should include a `mode: "self_hosted"` claim. At startup, the server checks:

```
if config.deployment_mode == "hosted" && license_claims.mode == "self_hosted":
    refuse to start — log error "license does not permit hosted mode"
```

This prevents a customer from switching to hosted mode to bypass `SelfHostedChecker`.

### Principle 7: Remote heartbeat for tamper detection

Even if a customer disables licensing (`enabled=false`), their server should still send a periodic heartbeat to our endpoint:

- The server process (not the license checker) pings `https://api.freerangenotify.com/v1/heartbeat` every 6 hours with the instance ID and license key.
- If our server stops receiving heartbeats, we flag the customer.
- This is a monitoring/alerting concern, not a hard block — but it deters tampering.
- The heartbeat call is baked into the server startup, not controlled by the `licensing.enabled` flag.

2. CLI calls `http://localhost:8080/v1/ops/subscriptions/renew` with `Authorization: Bearer ops:<secret>`
3. Server validates ops secret, performs action, returns result

### Phase 3: Build Tags + Revamp `frn install` for Self-Hosted Distribution

**New files (build-tag gating):**
- `internal/config/licensing_default.go` (`//go:build !selfhosted`) — `LicensingOverrides()` returns `nil` (hosted: config from file/env)

- `internal/config/config.go` — after unmarshalling, call `LicensingOverrides()` and apply if non-nil
- `internal/interfaces/http/routes/routes.go` — ops routes registration also gated by build tag (not just config value)
- `Makefile` — add `build-selfhosted` target: `go build -tags selfhosted -ldflags='-w -s' -o server ./cmd/server`

**CI/CD builds two sets of binaries:**
```
make build            → hosted server + worker (licensing configurable)
make build-selfhosted → self-hosted server + worker (licensing compiled in)
```

**Revamp `frn install`:**
- `cmd/frn/install.go` — rewrite to:
  - Prompt for ES URL/creds, Redis host/creds, license key
  - Validate license key against `https://api.freerangenotify.com/v1/license/verify` before proceeding
  - Download pre-built **self-hosted** `server` and `worker` binaries from CDN
  - Generate `config.yaml` with infra creds + license key only (no licensing on/off — that's baked in)
  - Generate systemd service files (`freerange-server.service`, `freerange-worker.service`)
  - Start services and run health check
  - Do NOT generate docker-compose.yml or .env

**New file:**
- `cmd/frn/systemd.go` — templates for systemd unit files

### Phase 4: Self-Hosted Remote Verification + Heartbeat

- Add `RemoteVerifier` that periodically calls our central server
- `SelfHostedChecker` uses `RemoteVerifier` alongside local JWT validation
- Fetch public key PEM from remote on startup instead of config file
- Add heartbeat goroutine in server startup (independent of licensing config)
- License JWT includes `mode` claim; server validates mode at startup
- Self-hosted binary ignores `FREERANGE_LICENSING_ENABLED` env var (always enabled)

### Phase 5: CDN + Binary Release Pipeline

- CI/CD builds `server` and `worker` as standalone Linux binaries (not in Docker)
- Publishes to CDN (e.g., `https://releases.freerangenotify.com/v0.2.2/linux-amd64/server`)
- `frn install` downloads from this CDN
- Signed checksums to verify binary integrity

## Security Summary

| Attack | Mitigation |
|---|---|
| Regular user calls ops endpoints | Ops endpoints require ops secret, not JWT. Users don't have the secret. |
| Regular user calls existing subscription CRUD | POST/PUT moved to ops routes. Only GET remains under JWT (read-only). |
| Self-hosted customer calls ops endpoints | Routes are not registered in self-hosted mode. 404. |
| Self-hosted customer creates fake subscriptions in ES | Self-hosted mode uses `SelfHostedChecker` which doesn't read ES subscriptions at all. |
| Self-hosted customer switches `deployment_mode` to `hosted` | Server validates `mode` claim in license JWT at startup. Refuses to start if mismatch. |
| Self-hosted customer sets `licensing.enabled=false` via env var | Build tag: `LicensingOverrides()` returns hardcoded `true`. Env var ignored. Compiled constant. |
| Self-hosted customer edits `config.yaml` to disable licensing | Build tag: config value overridden by `LicensingOverrides()` after unmarshal. Cannot change compiled code. |
| Self-hosted customer forges a license key | Remote verification against our server catches forged/revoked keys. |
| Self-hosted customer uses their own RSA key pair | Public key PEM fetched from our server at startup, not from customer's config. |
| Self-hosted customer gets binaries without valid license | `frn install` validates license with our server BEFORE downloading binaries. |
| Ops secret leaked | Rotate via config/env. Never stored in source code. |
| Someone intercepts ops secret over network | CLI only talks to localhost/internal network. Production uses TLS. |
| Customer rebuilds from source with checks removed | Legal protection (license agreement). Heartbeat detects missing instances. Accepted risk for open source. |

## What We Are NOT Doing

- No new admin role or super-admin user model (unnecessary complexity).
- No new JWT scopes (ops secret is simpler and more secure for machine-to-machine).
- No direct Elasticsearch writes from CLI (all actions go through backend APIs for consistency and audit).
- No subscription management UI for ops (CLI only — this is intentional for security).
